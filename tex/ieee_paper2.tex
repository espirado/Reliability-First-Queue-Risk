\begin{abstract}
Queue-delay predictors for GPU clusters are useful only if their probabilities are reliable: a ``70\% risk'' should occur roughly 70\% of the time. We present a reliability-first, open-source pipeline that treats probability \emph{calibration} as a primary evaluation criterion alongside discrimination and latency, and that integrates with production SLOs. The pipeline spans four heterogeneous environments (Amazon EKS, $1.2$M pod events; AWS ParallelCluster Slurm HPC, 555 GPU jobs with DCGM telemetry; the public Alibaba GPU trace; the public Google Borg trace), evaluates five model families with post-hoc isotonic calibration, and assigns each trained model to one of four operational tiers (warn / advisory / suggest / gate) via SLO-compliance tests. On Slurm, enriching submit-time features with aggregated DCGM telemetry raises Logistic Regression from AUROC $0.498$ (\textit{warn}) to $0.815$ (\textit{gate}) while cutting ECE from $0.066$ to $0.027$; Gradient Boosting reaches ECE $0.0018$ and qualifies for admission-gating. On Alibaba and Borg, calibration holds (ECE $\leq 0.031$) even when discrimination collapses---the appropriate behavior for an advisory predictor. Cross-cluster PSI on \texttt{pending\_ratio} between EKS and Slurm reaches $12.4$, motivating per-cluster recalibration; a temporal-ECE monitor catches drift at window seven of eight and triggers recalibration. We also ship a trace-calibrated discrete-event Slurm simulator that runs the full pipeline on $10{,}000$-job workloads under controlled perturbations, unit-testing the calibration-monitoring SLI: under a controlled input-distribution shift, ECE degrades $32$--$36\times$ while AUROC stays at $0.99$. The full pipeline, simulator harness, data samples, trained calibrators, and a single reproducibility notebook are archived on Zenodo.
\end{abstract}

\section{Introduction}
Submit-time queue-risk prediction has become a common tool in GPU-cluster operations: an ML model estimates the probability that a newly submitted job will wait longer than some SLO threshold, and downstream systems advise the user or adjust scheduling. For this workflow to be safe, the probabilities must be \emph{reliable}: a ``70\% risk'' should occur about 70\% of the time. Yet most published queue-risk models report only AUROC or accuracy, metrics that are invariant to monotone rescaling of probabilities and therefore silent on calibration. When an uncalibrated model is wired into an admission webhook, miscalibration becomes user-visible error: the system gates jobs the user believes are safe, or admits jobs the user believes are risky.

This paper describes \textit{reliability-first} queue-risk prediction as a scientific-software contribution. We treat probability calibration, drift detection, SLO compliance, and safe rollout as engineering deliverables on par with discrimination and latency, and we release the full pipeline (feature engineering, five model families, isotonic post-hoc calibration, the tier-qualification algorithm, a PSI-based drift monitor, and a sliding-window recalibrator) with a single Jupyter notebook that regenerates every figure in this paper.

\subsection{Contributions}
\begin{itemize}
    \item A \textbf{reproducible calibration benchmark} across four heterogeneous GPU-cluster environments (EKS, Slurm, Alibaba, Borg) spanning $\sim$1.2M job events and three scheduler families, with bootstrap 95\% CIs and McNemar significance tests on five model families.
    \item An empirical demonstration that \textbf{DCGM telemetry aggregated at submit time} rescues a linear model on Slurm: Logistic Regression with all features achieves ECE 0.027 and qualifies for admission-\textit{gate}, versus ECE 0.066 and \textit{warn}-only with submit-time features alone.
    \item A four-tier \textbf{operational qualification algorithm} that maps calibration and tail-behavior tests to concrete operational roles (warn / advisory / suggest / gate), so operators can pick a risk-appropriate integration without rereading the full evaluation.
    \item A \textbf{drift-to-recalibration pipeline} that combines PSI on submit-time features with a temporal-ECE monitor, triggering sliding-window isotonic recalibration before SLO alarms fire.
    \item A \textbf{controlled scaling experiment} (\S\ref{sec:sim}) built on a trace-calibrated discrete-event Slurm simulator, used as a reproducibility harness for the calibration pipeline. It (a) confirms that the bootstrap variance estimates from the 555-job trace are statistically conservative, (b) shows a median-threshold heuristic is structurally dominated even under the conditions most favorable to it, and (c) unit-tests that the PSI and temporal-ECE SLIs fire under controlled input-distribution shift while discrimination metrics remain intact.
    \item An \textbf{open artifact} (Zenodo DOI in \S\ref{sec:repro}) containing sample data, trained calibrators, the controlled-experiment harness, a heuristic-vs-ML baseline comparison, a decision-lead-time analysis, a literature-derived DCGM$\to$failure-mode mapping, the notebook, and IEEE \LaTeX{} source, designed for the ISS26 audience of research software engineers.
\end{itemize}

Modern GPU infrastructure has reached a point where throughput-only metrics are insufficient. Industry reports now prioritize \emph{goodput}, the successful-completion rate, over raw compute capacity~\cite{coreweave2025}, and analyses of production AI workloads identify queue delays and interruptions as a 15--25\% goodput tax in shared clusters. The three operational failure modes that drive this tax (queue congestion from head-of-line blocking by long jobs, resource fragmentation from stranded GPUs the scheduler cannot pack, and cascading delays where one slow node forces serial restarts of dependent jobs) are precisely the regimes in which a calibrated submit-time risk score gives operators a \emph{decision lead time} window to reroute or warn before resources are committed. Meta's LLaMA-3 training report~\cite{llama3} attributes 58.7\% of unexpected training failures to GPU hardware issues, and Modal's production data across 20{,}000+ GPUs~\cite{modal2025} documents systematic thermal, ECC, and HW\_SLOWDOWN events that cascade into scheduling disruption. PinDrop~\cite{pindrop2026} reports that 0.0024\% of machines begin to silently fail each quarter post-burn-in, making continuous telemetry-driven prediction rather than one-time deployment checks a sustainability requirement. Our pipeline responds to these findings by treating calibration quality and drift detection as deliverable outputs, not afterthought diagnostics.

\section{Background and Metrics}
\label{sec:bg}
\subsection{Calibration}
A classifier producing scores $p_i \in [0,1]$ on examples $(x_i, y_i)$ with $y_i \in \{0,1\}$ is \emph{calibrated} if $\Pr(Y = 1 \mid p = q) = q$ for every $q$. We estimate miscalibration via the Expected Calibration Error (ECE) with $B=15$ equal-mass bins,
\begin{equation}
\mathrm{ECE} = \sum_{b=1}^{B} \frac{|B_b|}{N}\, \left| \mathrm{acc}(B_b) - \mathrm{conf}(B_b) \right|,
\end{equation}
and the Maximum Calibration Error $\mathrm{MCE} = \max_b |\mathrm{acc}(B_b) - \mathrm{conf}(B_b)|$ for worst-bin behavior. We further report Murphy's three-way Brier-score decomposition~\cite{murphy1973} into \emph{reliability}, \emph{resolution}, and \emph{uncertainty}, which attributes differences in Brier score to genuine improvements in calibration vs. in ranking vs. in intrinsic difficulty of the task.

\subsection{Tail calibration}
Operational decisions typically fire at thresholds $\tau \in \{0.5, 0.6, 0.7, 0.8, 0.9\}$. We therefore report a \emph{tail-calibration gap} at each threshold---the absolute difference between predicted and observed positive rates among examples with $p \geq \tau$---which surfaces miscalibration concentrated in the high-risk tail even when whole-range ECE looks acceptable.

\subsection{Drift}
We monitor input drift with the Population Stability Index~\cite{siddiqi},
\begin{equation}
\mathrm{PSI} = \sum_{k=1}^{K} (q_k - p_k)\log(q_k / p_k),
\end{equation}
computed over $K=10$ equal-frequency bins of a reference distribution $p$ versus a current distribution $q$. Following the banking convention, $\mathrm{PSI} < 0.1$ indicates no material shift, $0.1 \leq \mathrm{PSI} < 0.25$ moderate shift, and $\mathrm{PSI} \geq 0.25$ severe shift. We report PSI at the feature level and temporal ECE at the prediction level.

\section{Methodology}
\label{sec:method}
Each subsection names the specific file or script involved so every number can be reproduced from the released repository.

\subsection{Data sources and collection}
Each environment ships a public CSV sample under \texttt{data/samples/} so reviewers can rerun without raw-trace access. All models are trained and evaluated within a single environment; cross-environment transfer is studied in~\S\ref{sec:drift} via PSI on the EKS$\to$Slurm pair.

\textbf{EKS (K8s).} $1.2$M pod-event trace from a production Amazon EKS cluster via the K8s informer API, joined with cAdvisor and NVIDIA DCGM telemetry (identifiers hashed). Sub-sampled evaluation split \textit{EKS-Dec} ($n_{\text{test}}{=}103$) at \texttt{data/samples/eks\_dec\_sample.csv}.

\textbf{Slurm HPC.} 555-job trace on an AWS ParallelCluster Slurm deployment (four GPU nodes, controlled load test). \texttt{code/features/feature\_engineering.py} joins \texttt{sacct} queue state with per-node DCGM snapshots at submit timestamps; split at \texttt{data/samples/slurm\_sample.csv}.

\textbf{Alibaba.} Public cluster-trace-gpu-v2020 and v2023~\cite{weng2022mlaas,alibaba-2023-fgd}. Raw queue-state snapshots are not published, so \texttt{pending\_ratio}/\texttt{queue\_depth} are reconstructed by trace-replay: for each submit timestamp we count concurrently-pending jobs. Replay logic lives in the same feature-engineering file; split at \texttt{data/samples/alibaba\_sample.csv}.

\textbf{Google Borg.} 2019 trace at \texttt{data/samples/borg\_sample.csv}. Borg is CPU-dominant; we keep it as a negative control that exercises graceful degradation (low AUROC, still-calibrated probabilities).

\subsection{Feature engineering}
\label{sec:features}
Two feature families are computed in \texttt{code/features/feature\_engineering.py}.

\emph{Submit-time, scheduler-agnostic.} Ten queue-state and topology features available at admission time: pending-to-capacity ratio, normalized queue depth, fragmentation score, congestion score, total pending and pending-GPU counts, running-GPU and GPU-node counts, and an encoded job type. These are computable from any scheduler that exposes a queue state.

\emph{DCGM telemetry (Slurm and EKS).} Per-cluster aggregates snapshotted at each submit timestamp, organized into four families:
(i)~\emph{Thermal anomalies}: GPU temperature mean/max. Values above $80^{\circ}$\,C warn of HW\_SLOWDOWN trips at $85^{\circ}$\,C (A100) / $87^{\circ}$\,C (H100)~\cite{llama3,modal2025}, where cards throttle before failing active health checks.
(ii)~\emph{Utilization dispersion}: GPU utilization mean/std/max plus CV and inter-quantile range, capturing straggler-GPU patterns linked to NCCL all-reduce tail latency~\cite{pindrop2026}.
(iii)~\emph{Power/memory spread}: power-draw mean/max, memory-used mean/max, derived power-efficiency and memory-pressure ratios. High power variance is the signature of HW\_POWER\_BRAKE on badly-provisioned racks~\cite{modal2025}.
(iv)~\emph{Cross-feature dispersion}: cluster-wide P90--P10 spreads, motivated by Modal's passive-healthcheck taxonomy.
Unlike Modal's active checks (which can take roughly an hour and require exclusive node access), these snapshots are read continuously from running DCGM exporters at zero scheduling cost.

\subsection{Label construction}
The binary label \texttt{label\_long\_wait} is 1 if the observed wait exceeds the per-dataset 90th percentile. P90 thresholds are precomputed and stored alongside the features (column \texttt{wait\_threshold\_p90}). Positive rates (Table~\ref{tab:datasets}) span $9.8$--$34.0\%$, so every reported result is a class-imbalanced classification.

\subsection{Train/test protocol}
5-fold stratified CV (seed 42) for model selection; a held-out test split for reported numbers. Confidence intervals are $1{,}000$-resample percentile bootstrap (stored in \texttt{bootstrap\_confidence\_intervals.csv}). A temporal hold-out (last 20\% by submit timestamp) is reserved for drift analysis to avoid CV leakage. Test-set sizes are reported per environment.

\subsection{Model zoo and hyperparameters}
Five model families chosen for ubiquity in the scientific-Python ecosystem: Logistic Regression (L2, $C{=}1.0$), Random Forest ($300$ trees, depth $12$), Gradient Boosting ($200$ trees, lr $0.05$, depth $4$), XGBoost (same shape, \texttt{scale\_pos\_weight} from positive rate), LightGBM ($64$ leaves, lr $0.05$). A single hyperparameter configuration is used across all environments, testing whether a ``stock'' ML setup produces usable models across schedulers.

\subsection{Calibration}
Post-hoc isotonic regression on an out-of-fold validation split, wrapped in \texttt{CalibratedClassifierCV(cv='prefit')}. We choose isotonic over Platt for two reasons: (a) raw-score reliability diagrams on Alibaba and Slurm show non-monotone miscalibration Platt's sigmoid cannot model; (b) the small Slurm split still leaves enough samples per bin for a stable piecewise-constant fit. Platt scaling ships as a sensitivity check in the notebook.

\subsection{Evaluation metrics}
We report seven metrics, each with a distinct role:
\begin{itemize}
    \item \textbf{AUROC} and \textbf{AUPRC} for discrimination;
    \item \textbf{Brier score} for joint quality, decomposed via Murphy~\cite{murphy1973} into reliability, resolution, and uncertainty;
    \item \textbf{ECE} with $B=15$ equal-mass bins for global calibration;
    \item \textbf{MCE} for worst-bin calibration;
    \item \textbf{tail calibration gap} at $\tau \in \{0.5,0.6,0.7,0.8,0.9\}$ for operational-band behavior;
    \item \textbf{McNemar's test} with Holm--Bonferroni correction for pairwise model comparisons (\texttt{artifacts/mcnemar\_5\_models.csv}).
\end{itemize}

\subsection{Tier qualification algorithm}
\label{sec:tiers}
We assign each trained-and-calibrated model to one of four operational tiers:
\begin{enumerate}
    \item \textit{warn} (informational only): minimum requirement ECE $\leq 0.10$;
    \item \textit{advisory} (surface in UI with no action): ECE $\leq 0.05$ and false-action rate at threshold $0.5$ below $20\%$;
    \item \textit{suggest} (auto-suggest alternate queue or GPU count): ECE $\leq 0.05$, MCE $\leq 0.15$, tail-gap at $\tau = 0.7$ below $0.05$;
    \item \textit{gate} (policy-level admission): ECE $\leq 0.05$, MCE $\leq 0.10$, tail-gap $\leq 0.035$ at every $\tau$, and false-action rate at $\tau = 0.7$ below $5\%$.
\end{enumerate}
A model qualifies for tier $k$ iff it satisfies \emph{all} criteria at tier $k$ \emph{and} every lower tier. This is computed in \texttt{code/evaluation/tier\_qualification.py}; the resulting assignments are stored with the experiment JSON.

\subsection{Drift detection and sliding-window recalibration}
\label{sec:recalib-method}
The recalibrator is a stateful sidecar that tracks two rolling-window signals: \emph{(a) feature drift} via PSI against a frozen deployment reference, and \emph{(b) prediction drift} via temporal ECE on a rolling window of labeled predictions.

\textbf{Window size and cadence.} Default $7$ days for production or $250$ jobs for low-volume HPC (both exposed as YAML knobs). Windows slide by $1$ day or $50$ jobs, so detection latency is one window step, not one full window.

\textbf{Trigger logic.} Refit fires when any feature's $\mathrm{PSI}{>}0.1$ (data drift, surfaces before labels arrive) or windowed $\mathrm{ECE}{>}0.07$ (concept drift, surfaces once outcomes are observed). We use $0.07$---not the $0.05$ \textit{gate} SLO bound---so the refit fires \emph{before} on-call would be paged.

\textbf{Refit procedure.} On trigger, the classifier is frozen, a fresh isotonic regression is fit on the most recent valid window, and the new calibrator is written to a versioned store with one-click rollback. Refits are $O(N \log N)$ in window size and complete in milliseconds.

\textbf{Rate limiting.} A $24$-hour cooldown between refits prevents oscillation; a trigger inside the cooldown pages on-call rather than auto-refitting. The full logic lives in \texttt{code/ops/recalibration\_loop.py} and is portable to a K8s controller or Slurm \texttt{job\_submit} plugin without pulling in the training stack.

\subsection{DCGM telemetry rescues linear models on Slurm}

\section{Controlled Scaling Experiment via Trace-Calibrated Discrete-Event Simulation}
\label{sec:sim}
The real Slurm measurement in~\S\ref{sec:bench} is a 555-job AWS ParallelCluster load test: intentionally small and intentionally auditable. The four real datasets exercise the empirical claims; we use a separate \emph{controlled scaling experiment} to stress-test the software-engineering properties of the pipeline (reproducibility at scale, the structural sufficiency of a heuristic baseline, and whether the calibration-monitoring SLIs fire under controlled input perturbation). The harness is a trace-calibrated discrete-event Slurm simulator following the tradition of BatSim~\cite{batsim2017}, AccaSim~\cite{accasim2018}, and the Lublin--Feitelson workload model~\cite{lublin2003}. It is not additional empirical evidence about real GPU clusters or A100/H100 hardware; it is a reproducibility harness for the operational pipeline.

\subsection{Scope and limits of the controlled experiment}
\label{sec:sim-scope}
We are explicit about what this experiment can and cannot demonstrate.

\textbf{What it can demonstrate.} (1) end-to-end deterministic execution at $10{,}000$ jobs from a single seed; (2) that the $N{=}555$ bootstrap envelopes in~\S\ref{sec:bench} are statistically conservative (CIs tighten predictably at scale); (3) that a median-threshold heuristic is structurally dominated by a calibrated ML model on actionable precision \emph{even in the regime most favorable to it}; (4) that the PSI and temporal-ECE SLIs of~\S\ref{sec:drift} fire under a controlled input-distribution shift when discrimination remains intact.

\textbf{What it cannot demonstrate.} Because the harness uses Slurm-backfill mechanics under full observability, wait time is by construction a near-deterministic function of submit-time queue state, so AUROC values approach $1.00$ as a property of the experiment, not a claim about real-cluster predictability. DCGM aggregates are sampled as marginal distributions per profile and are not causally linked to wait time; the cross-profile result in~\S\ref{sec:hwshift} is a unit-test for the calibration-monitoring SLI under controlled shift, not a prediction about A100/H100 hardware. The four real datasets in~\S\ref{sec:bench} are the empirical evidence; the controlled experiment is a software-engineering robustness study.

\subsection{Experimental harness design}
The harness (\texttt{slurm\_simulator.py} in the released repository) is seeded, deterministic, and calibrated on the real 555-job trace. Inter-arrival times are resampled from the empirical IAT distribution (mean 48.9\,s, median 20\,s, max 1{,}141\,s) with an \texttt{arrival\_scale} multiplier for stress variants. Job-type mixture, per-type GPU-request distribution, and per-type service-time distribution are drawn from the real trace. Scheduling follows Slurm conservative backfill: small jobs may be promoted ahead of the queue only if doing so does not delay the head-of-line reservation. Submit-time features (\texttt{pending\_ratio}, \texttt{queue\_depth\_norm}, \texttt{fragmentation\_score}, \texttt{congestion\_score}, and others) are captured before admission, preserving the non-leakage contract from~\S\ref{sec:method}. DCGM aggregates (temperature, power, utilization dispersion) are drawn from two parameterized marginal profiles: \emph{profile-V}, calibrated to the v100/t4 envelope of our real trace, and \emph{profile-A}, whose marginals track the published telemetry envelope of the A100/H100 generation reported in Meta LLaMA-3~\cite{llama3}, Modal's 20{,}000-GPU fleet~\cite{modal2025}, and PinDrop~\cite{pindrop2026} (mean utilization 55\% vs.\ 25\%; max temperature $78^{\circ}$\,C vs.\ $55^{\circ}$\,C; power 310\,W vs.\ 85\,W). The two profiles form a controlled distribution-shift pair; they are not claims of equivalence with measured A100/H100 telemetry.

\subsection{Three controlled scenarios and pipeline-at-scale results}
We run three $10{,}000$-job controlled scenarios: \emph{baseline} (profile-V, real-matching arrivals), \emph{heavy} (profile-V, $2\times$ arrival rate), and \emph{shift} (profile-A marginals, real-matching arrivals). After admission by the backfill policy, $7{,}709$, $4{,}998$, and $8{,}026$ jobs respectively enter the evaluation set (the remainder are still pending at the horizon). All three scenarios retain the 9.9\% empirical positive rate from the real trace because the P90 wait threshold is recomputed per scenario. Table~\ref{tab:sim-scale} reports LR, RF, and GB with isotonic calibration on each scenario, with bootstrap-95\% AUROC CIs. As anticipated in~\S\ref{sec:sim-scope}, AUROC approaches $1.00$ in all three scenarios because the experiment's submit-time feature set is, by construction, near-sufficient under the harness's full-observability conditions. The relevant reading of Table~\ref{tab:sim-scale} is not the AUROC magnitude but its low variance: the bootstrap intervals collapse to $\pm 0.003$, confirming that the wider intervals at $N{=}555$ in~\S\ref{sec:bench} are statistically conservative. \subsection{Structural insufficiency of a heuristic baseline}
A reasonable critique of any ML approach to queue prediction is that a simple threshold rule (e.g., \emph{if queue depth exceeds the cluster median, predict a long wait}) might do as well, especially given how cleanly queue depth correlates with wait. The controlled experiment is well-suited to settling this question because it lets the heuristic operate in the regime that is most favorable to it: one in which queue depth is, by construction, a near-sufficient statistic for wait. Table~\ref{tab:sim-heur} compares the heuristic against the calibrated GB model in all three controlled scenarios and on the real trace. The heuristic achieves high AUROC (0.55--0.99) because queue depth and wait covary, yet its precision never exceeds 0.21, meaning roughly four out of five high-risk flags would be false alarms if acted on. The calibrated GB model, in the same test sets, lands in \emph{gate} tier with AUPRC 0.98--1.00 in the controlled experiment and 0.32 on the $N{=}555$ real trace, still over three times the 0.10 precision of the heuristic on the real data.

\emph{Why does heuristic precision drop from $0.20$ controlled to $0.098$ real?} A single-threshold rule's precision is bounded above by the positive rate observed when the threshold fires. Under the controlled harness, the median rule fires on the upper half of the queue-depth distribution, of which the wait-P90 set is a near-uniform subset; precision therefore concentrates near the global positive rate ($0.099$, doubling to $\approx 0.20$ inside the firing half). The real trace adds confounders the rule cannot see---job-type mix, GPU-class fragmentation, transient operator actions on partition limits---so the same threshold lights up many short-wait jobs, and precision drops. The calibrated GB model exploits these confounders through the features in Table~\ref{tab:featimp}, which is why the gap between the two methods \emph{widens} in the noisier real regime. The conclusion is structural: discrimination is cheap, but actionable precision requires model structure beyond a single threshold.

\subsection{Monitoring-SLI unit test under controlled distribution shift}
\label{sec:hwshift}
The goal of this experiment is not to predict A100 or H100 behavior (we have no real telemetry from such hardware in this work) but to unit-test the calibration-monitoring SLI design from~\S\ref{sec:drift}: when an input distribution shifts, do PSI and temporal-ECE fire even if discrimination metrics remain green? Because DCGM aggregates inside the harness are sampled from independent marginal distributions per profile, this is a clean, isolated test of the SLI's response to controlled input perturbation, not a claim about hardware-specific failure modes. We train the full LR, RF, and GB pipelines on the baseline (profile-V) trace and test on the shift (profile-A marginals) trace, with no recalibration in between. Calibration moves under input perturbation; discrimination does not. This is the failure mode that motivates treating ECE, tail-gap, and PSI as primary SLIs, and the experiment confirms that the SLIs catch it under the controlled shift the harness is designed to produce.

\section{Integration Pattern and Operational Case Study}
\label{sec:ops}
A reliability-first predictor is only useful if operators can deploy it without fear. This section describes the integration pattern we ship, a short case study of running the pipeline on our own Slurm cluster for six weeks, and the operational lessons that pattern encodes.

The calibrated predictor integrates as an \emph{advisory} service that sits beside, not inside, the scheduler:

\textbf{Kubernetes.} A mutating admission webhook scores incoming Pods and writes a \texttt{scheduling-advisory/risk} annotation. Consumers (dashboards, team bots, or downstream controllers) read the annotation; no scheduling decision is blocked by the predictor.

\textbf{Slurm.} A \texttt{job\_submit} Lua plugin calls the local scoring service and either adds a comment on the job (\textit{advisory} tier) or suggests an alternate partition (\textit{suggest} tier). \textit{Gate} behavior is only enabled per-namespace after operator opt-in with a feature flag and rollback path.

\textbf{Shadow $\to$ advisory $\to$ gate.} All deployments start in shadow mode (the predictor logs but does nothing), graduate to advisory when the weekly-ECE SLO holds for four consecutive weeks, and only reach \textit{suggest}/\textit{gate} after a documented canary. The sliding-window recalibrator runs in all modes so that calibration never drifts silently.

\subsection{Case study: six weeks on an internal Slurm cluster}
We deployed the \textit{advisory}-tier integration on the Saint Peter's Slurm ParallelCluster for six weeks. Three observations stand out. (i)~The dominant drift source was \emph{topology}, not workload: replacing a failing A10G node with a T4 shifted \texttt{total\_pending}/\texttt{pending\_gpus} immediately, PSI crossed $0.1$ within an hour, and the sliding-window recalibrator stabilized weekly-ECE below $0.02$ in the next window without human intervention. (ii)~Operators reacted to the false-action rate at $\tau=0.7$ rather than ECE alone: ECE $0.05$ with tail-gap concentrated at $\tau=0.85$ produced visibly wrong advisories for a small group of users, and demoting \textit{suggest}$\to$\textit{advisory} until the next refit restored trust. (iii)~The K8s mutating webhook and the Slurm \texttt{job\_submit} plugin converged on the same JSON scoring protocol, so the recalibration loop, dashboards, and on-call runbooks were shared across both schedulers.

The operator-facing knobs are intentionally few: the recalibration trigger (\texttt{psi\_threshold}, \texttt{weekly\_ece\_threshold}), the tier assignment (a human-readable YAML), the advisory-text template, and the per-namespace feature flag that promotes a model from shadow to advisory to suggest. Everything else is either pipeline-internal (covered by the CI test matrix) or documented as out of operator scope (training hyperparameters). A small operator surface is, in our experience, the single most important reproducibility property for an on-call-facing ML system. \section{Related Work}
\textbf{Calibration methods.} Platt scaling~\cite{platt1999}, isotonic regression~\cite{zadrozny2002}, and modern deep-network calibration~\cite{guo2017calibration} form the methodological basis of our post-hoc calibrator. Density-Aware Calibration~\cite{dac} and Parameterized Temperature Scaling~\cite{pts} are alternatives we include as sensitivity checks.

\textbf{Operational ML systems.} The ``hidden technical debt'' critique~\cite{sculley2015} motivates our treatment of calibration and recalibration as first-class components rather than as afterthoughts.

\textbf{Cluster scheduling.} Borg~\cite{borg2015}, Firmament~\cite{firmament2016}, and learning-based schedulers such as Decima~\cite{decima2019} establish the scheduling substrate; our contribution is orthogonal, providing reliable \emph{probabilistic advice} that any of these schedulers can consume.

\textbf{GPU hardware reliability.} Meta LLaMA-3~\cite{llama3}, Modal's 20{,}000-GPU healthchecking experience~\cite{modal2025}, and PinDrop~\cite{pindrop2026} establish the empirical basis for DCGM dispersion features: hardware health anomalies are common and cascade into scheduling disruption.

\textbf{Industry practice.} CoreWeave's goodput framing~\cite{coreweave2025} and the convergence of Kubernetes + Slurm as the dual orchestration standard motivate our unified feature schema across both environments.

\section{Discussion and Limitations}
\textbf{Calibration holds where discrimination fails.} On Borg, AUROC is near chance yet ECE stays below $0.014$. This is not a contradiction: calibrated ``I don't know'' is a useful signal. Operators can configure the advisory to surface only predictions whose Brier resolution exceeds a floor, so calibrated indifference does not crowd the UI.

\textbf{DCGM rescues weak models.} The Slurm submit-only LR $\to$ submit+DCGM LR delta (AUROC $0.498 \to 0.815$, tier \textit{warn} $\to$ \textit{gate}) shows that investment in telemetry yields more reliability than investment in model complexity. For operators we recommend deploying the DCGM exporter before tuning hyperparameters.

\textbf{Cross-environment transfer.} Each model in~\S\ref{sec:bench} is trained and tested within a single environment. The cross-environment behavior we do study is the EKS$\to$Slurm pair in~\S\ref{sec:drift}: PSI on \texttt{pending\_ratio} reaches $12.4$, two orders of magnitude above the $0.25$ severe-shift threshold, so direct cross-cluster transfer of a trained model is not viable. The pipeline's transfer story is therefore not a single global model but a per-cluster instantiation that shares the calibration-monitoring SLI design. A controlled-shift unit test (Table~\ref{tab:cross}) shows the SLIs catch this regime even when AUROC stays high.

\textbf{A100/H100 validation feasibility.} Table~\ref{tab:dcgm-modes} is literature-derived planning guidance, not validated A100/H100 results. A targeted validation is feasible with the existing pipeline: the DCGM aggregator accepts standard exporter feeds, so a small ($\geq 24$\,h) A100/H100-partition trace would suffice to test whether the thresholds in Table~\ref{tab:dcgm-modes} trigger on the literature-cited failure modes. We document this as the natural follow-on; the required instrumented A100/H100 fleet (with HW\_SLOWDOWN and HBM-OOM surfaced via DCGM) is outside the v100/t4 testbed used here.

\textbf{Other limitations.} The real Slurm trace is modest ($n{=}555$), reflecting the AWS ParallelCluster load-test it came from; every result is marked real-measurement or controlled-experiment. The four real datasets in~\S\ref{sec:bench} form the empirical base; the controlled experiment is a robustness harness whose AUROC $\to 1.00$ is a property of the harness, not a real-cluster claim. The Alibaba \texttt{pending\_ratio} is trace-derived; the EKS test split ($n_{\text{test}}{=}103$) is why we report CIs. Tier thresholds in~\S\ref{sec:tiers} are operationally derived and treated as deployment hyperparameters.

\section{Conclusion}
We have presented a reliability-first, reproducible pipeline for GPU-cluster queue-risk prediction. The pipeline treats probability calibration, drift detection, SLO compliance, and safe rollout as engineering objectives on par with discrimination and latency. On four real environments spanning $\sim$1.2M job events, calibrated predictors qualify for concrete operational tiers, DCGM telemetry rescues underspecified linear models on Slurm, and the sliding-window recalibrator catches drift before SLO alarms fire. We complement the empirical evaluation with a trace-calibrated controlled scaling experiment that serves as a reproducibility harness for the pipeline: it confirms bootstrap variance estimates, shows that a heuristic baseline is structurally insufficient even under model-favorable conditions, and unit-tests the calibration-monitoring SLI design under controlled input-distribution shift. The released artifact (data samples, trained calibrators, controlled-experiment harness, notebook, and IEEE \LaTeX{} source) lets an ISS26 audience reproduce every figure in this paper and port the integration pattern to their own Kubernetes or Slurm clusters.

\section{Reproducibility}
\label{sec:repro}
\textbf{Single-command reproduction.} The repository ships with a single notebook, \texttt{notebooks/reproducibility.ipynb}, that executes the entire pipeline on subsampled data bundled in \texttt{data/samples/} and regenerates every figure in this paper. A user can \texttt{git clone}, \texttt{conda env create -f environment.yml}, and \texttt{jupyter lab notebooks/reproducibility.ipynb} without access to any external data.

\textbf{Environment pinning.} Python $3.11$ and dependency versions are pinned in both \texttt{environment.yml} (conda) and \texttt{requirements.txt} (pip), with a tested install on Linux and macOS. A \texttt{Binder} badge in the README provides a zero-install launch path for reviewers.

\textbf{Versioned artifacts.} Trained models, isotonic calibrators, bootstrap CI CSVs, and the per-experiment JSON results are stored under \texttt{artifacts/} and referenced from the paper tables. The CI pipeline recomputes the artifact hashes on every merge so drift in the code surfaces as a failed test.

\textbf{Release.} All code, sample data, trained calibrators, and this paper's \LaTeX{} source are released under the MIT License at \url{https://github.com/espirado/Reliability-First-Queue-Risk} and archived on Zenodo (DOI to be inserted after the v1.0.0 archive is minted).

\section*{Acronyms}
The following acronyms appear in the paper.
\begin{description}\setlength{\itemsep}{0pt}
\item[GPU] Graphics Processing Unit.
\item[SLO] Service-Level Objective: a target value or range for a service-level indicator (e.g., weekly ECE $\leq 0.05$).
\item[SLI] Service-Level Indicator: the metric that an SLO is defined over (e.g., weekly ECE, tail-gap at $\tau$).
\item[ECE] Expected Calibration Error: weighted-average gap between predicted confidence and observed frequency over $B$ bins (\S\ref{sec:bg}).
\item[MCE] Maximum Calibration Error: worst-case bin gap.
\item[AUROC] Area Under the Receiver-Operating-Characteristic curve.
\item[AUPRC] Area Under the Precision--Recall curve.
\item[DCGM] NVIDIA Data Center GPU Manager: the telemetry exporter used for thermal, power, and utilization aggregates (\S\ref{sec:features}).
\item[PSI] Population Stability Index: a divergence measure for input drift (\S\ref{sec:bg}).
\end{description}
